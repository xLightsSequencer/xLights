## The iOS networking / media boundary (be precise)

What iOS **can** do, and the iPad already exercises: HTTP/HTTPS POST/GET to controllers and proxies (every vendor handler), UDP broadcast discovery (`FPP/ArtNet/Twinkly/Pixlite16/DDP PrepareDiscovery`), and live sACN/ArtNet/DDP/E1.31 frame output via the real `OutputManager` (`startOutput`/`outputFrame:`). All upload protocols in this theme are HTTP file pushes — iOS handles them fine. Local-interface enumeration (for `ForceLocalIP` binding) is allowed under the sandbox.

What iOS **cannot** do, and where the line falls:
- **Raw serial / RS485 / USB-DMX output** — no host serial port. A serial controller record is still editable (and uploadable to an FPP-proxied serial port), but the iPad can never be the live serial output device. Already correctly handled.
- **Raw removable-drive enumeration / write** — there is no API to list/write an SD/USB card. This is the core reason the **HinksPix Export** SD-card/HSEQ/AU/schedule workflow is out of scope. HinksPix *network* config upload already works via the single-controller path.
- **Raw ICMP ping** — needs an entitlement xLights won't carry; use TCP/HTTP reachability probes instead (desktop's "ping" is largely HTTP/socket probes anyway).
- **FFmpeg encode** — not in the iPad lib; only relevant here for AU audio creation in HinksPix export (another reason that stays desktop-only).

## Approach

1. **Reuse shared core, add SwiftUI + bridge.** Almost every missing row is a new SwiftUI sheet/inspector section plus an `NS_SWIFT_NAME`'d wrapper on `XLSequenceDocument` over an existing `OutputManager` / `ControllerEthernet` / `Output` / `FPP` / `BaseController` method. No new protocol or upload algorithm is required.
2. **Funnel mutations through the view model.** All controller edits already set `hasUnsavedLayoutChanges` and persist via `saveLayoutChanges` → `iPadRenderContext::SaveLayoutChanges` → `OutputManager::Save()` + `RecalcStartChannels`. New setters must follow the same path so the dirty/save/recalc invariant holds in one place.
3. **Descriptor-driven property rows.** The controller property pane is descriptor-driven (`controllerPropertiesForName:` emits typed int/double/bool/enum/string rows; `setControllerProperty:` dispatches setters). The largest gap — protocol-specific Ethernet output config — is *new descriptor rows + matching setter branches*, not new UI plumbing.
4. **Touch idioms over desktop ports.** Bulk upload and bulk selection become a SwiftUI checklist sheet with an on-screen Cancel (no ESC). The destination-IP prompt is an inline sheet, not a modal `wxTextEntryDialog`. Live status is a SwiftUI timer feeding a per-row indicator, not a background ping thread.
5. **Gate by open-source firmware, deliberately (product decision — see below).** Upload / FPP-Connect / Visualize affordances stay gated to `caps.openSourceFirmware`, which admits exactly **FPP, WLED, and ESPixelStick** (the three `resources/controllers/*.xcontroller` configs declaring `<OpenSourceFirmware>` — verified). Keep that gate on every new upload/visualize entry point. Proprietary controllers are a deferred, likely IAP-gated future tier, not a port-now item.

## Open-source-firmware scope (product decision)

**For now the iPad app limits controller upload, FPP Connect, and the Visualize wiring view to open-source-firmware controllers only: FPP, WLED, and ESPixelStick.** This is a deliberate scope + monetization choice, **not** a platform limitation — iOS can HTTP-upload to any vendor, and all 14 vendor handlers already compile from shared `src-core/`. The three open-source vendors are exactly the `.xcontroller` configs that declare `<OpenSourceFirmware>`, and the live `caps.openSourceFirmware` gate (`XLSequenceDocument.mm:11779`) already reflects that set.

Implications and nuances:
- **General upload + Visualize gate** = `caps.openSourceFirmware` → FPP / WLED / ESPixelStick. WLED uploads through its own `WLED` `BaseController` HTTP path.
- **FPP Connect (multisync sheet) is narrower:** it admits only FPP + ESPixelStick *fppTypes* (`XLSequenceDocument.mm:14127-14134`, "iPad scope: FPP + ESPixelStick only"). WLED is not an FPP-protocol device, so it is configured/uploaded via the controller property pane + its own upload action, not the FPP Connect sheet. CTL-7's immediate-output path and CTL-11's FSEQ picker are therefore reachable only for FPP/ESPixelStick.
- **Proprietary / closed-firmware controllers** (Falcon V4/V5, Genius, HinksPix-network, PowerDMX, Pixlite/Advatek, SanDevices, J1Sys, …) **MAY be supported in the future** but are **low priority, and will most likely be enabled only via an in-app purchase / subscription tier.** Until then their *upload*, *FPP-Connect codec*, *HTTP-scan discovery*, and *Visualize* affordances are filtered/disabled (show a "Use desktop xLights for this controller" hint rather than failing silently). This is distinct from the genuinely platform-blocked pieces (HinksPix SD-card authoring, raw serial/DMX output) which stay hard out-of-scope (see `99`).
- Several work items below (CTL-7 immediate-output, CTL-11 FSEQ picker, **CTL-21 proprietary HTTP-scan discovery**, CTL-20 PixelTest network output) only matter for the *open-source* set today; the proprietary half of each rides the future IAP tier.

- **HinksPix Export dialog (SD-card HSEQ/AU/schedule authoring)** — primary output is writing HSEQ/playlists/schedule to a removable SD/USB card; iOS has no raw removable-drive enumeration/write API, and HinksPix uses proprietary firmware outside the open-firmware scope. (Network config upload still works via CTL-18.)
- **HinksPix Mode/Drive selection, HSEQ+AU creation, playlists/schedule editor, Export-to-SD vs Upload** — all dependents of the SD-card/proprietary-controller workflow; AU audio creation also needs the FFmpeg encode path the iPad lib doesn't build.
- **Bulk upload ESC cancel** — no physical ESC in the touch idiom; replaced by the on-screen Cancel button on CTL-3's bulk sheet.
- **Live serial / USB-DMX output** — no host serial port on iOS; the serial controller *record* is editable and uploadable to an FPP-proxied serial port, but the device can't be the live serial driver. (PixelTestDialog **serial/DMX** test output is likewise blocked — but the **network** test-pattern slice over sACN/ArtNet/DDP/E1.31 is feasible and is now in-plan as **CTL-20**, Phase 2 / XL, gated on the `TestFunctions`→channel-frame-generator core extraction.)
- **FPP Connect proprietary-codec upload paths (Slice E)** — the three closed-firmware HSEQ-style codec branches (Falcon V4/V5, Genius, PowerDMX in `FPPConnectDialog.cpp:1336-1404`) target devices outside the open-firmware scope; iPad filters those `fppType` rows out of the sheet (or disables them with a "Use desktop xLights" hint) rather than implementing the vendor codecs. See CTL-7 / CTL-11. *(Part of the deferred proprietary-controller tier above — could be unlocked behind the future IAP tier, though the vendor codecs are also genuine reimplementation work.)*

## Risks / open questions

- **CTL-1 scope creep.** The Ethernet output property set differs per protocol and the desktop logic lives in wx `OutputPropertyAdapters`. Confirm how much validation (universe-range collision, auto-size interplay with `RecalcStartChannels`) must be reproduced in the bridge vs. already enforced inside the `Output`/`ControllerEthernet` setters. Land E1.31/ArtNet first, then DDP, to de-risk.
- **CTL-8 ping fidelity.** Without ICMP, TCP/HTTP reachability may show "offline" for controllers that are up but firewall HTTP — acceptable, but the LED semantics should say "HTTP-reachable," not "pingable." Decide where the probe logic lives (core vs bridge) before CTL-9 commits to a column.
- **CTL-12 ordering.** Auto-upload-on-output-enable must run *after* `RecalcStartChannels` and proxy validation (CTL-4) and must not block the output toggle UI — run async with a non-fatal failure surface.
- **CTL-17 ZCPP extraction.** `SetZCPPPort`/`SetZCPPExtraConfig`/`SetModelData` are in `src-ui-wx` TabSetup, not core. Extraction crosses the core/UI boundary — verify it can move to `src-core` without dragging wx types (it must, to satisfy `check_core_include_boundaries.sh`). Given ZCPP's niche status, this is the right candidate to defer last.
- **CTL-18 HinksPix firmware flash** is HTTP-based and feasible, but firmware flashing over a flaky network is risky — gate behind a clear confirmation and verify `FirmwareSupportsUpload` before offering it.
- **Minleon ForceLocalIP** (Area 18 partial) — confirm local-interface enumeration / the desktop "flip" local-IP binding works under the iOS sandbox; needed for both CTL-1's Force-Local-IP row and Minleon upload.
- **CTL-15 sort silently re-channels (decided — gate it).** Controller list order drives start-channel assignment; phase-j / the master tracker dropped sort precisely because a stray tap re-channels every model with no recovery. The resolution is firm: sort must be gated behind a confirm + automatic `RecalcStartChannels` + an undo snapshot. Do not reintroduce a one-tap sort under any "quick win" framing.
- **CTL-20 core extraction is the real cost.** The `TestFunctions`→channel-frame-generator + `ChannelTracker` overlap-merge live in wx `PixelTestDialog` and must move to `src-core` wx-free (passing `check_core_include_boundaries.sh`) before the SwiftUI panel exists. Scope/verify that extraction first; the SwiftUI + selection-tree UI is comparatively cheap. Serial/DMX test output stays out of scope regardless.
- **CTL-23 fault-notification probe budget.** The mid-show reachability check must reuse CTL-8's TCP/HTTP probe, run off the playback thread, and back off so it never stalls frame output; a controller that is up-but-HTTP-firewalled will false-positive as "faulted" (same caveat as CTL-8's LED). Decide the debounce / consecutive-miss threshold before surfacing a banner.
