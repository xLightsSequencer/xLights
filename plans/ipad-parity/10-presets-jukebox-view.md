# iPad Parity Plan — Effect Presets, Jukebox & View / Window Management

**Scope:** Gap-analysis Section 23 (Jukebox, Effect Presets & Effect Tree) and the portable subset of Section 4 (View / Windows / Perspectives). The headline is a **persistent, hierarchical Effect Preset library** persisted to the rgbeffects `<effects>` node (today iPad presets are session-only, flat, single-effect), plus a **Jukebox equivalent**. Most of Section 4 (wxAUI perspectives/docking) is genuinely out-of-scope; only a handful of grid/dock toggles are portable.

> **PRE-1 keystone — ✅ LANDED 2026-06-01 (build-verified on iPad-lib + desktop; runtime round-trip/apply pending on-device).** Implemented via a worktree agent + central merge: `EffectPresetManager` relocated to wx-free `src-core/effects` (desktop + iPad share it; `EffectTreeDialog`/`EffectsGrid::Paste` stay desktop-only); `iPadRenderContext` owns the manager and loads/saves `<showFolder>/xlights_effectpresets.json` (+`.jbkp`, desktop format → cross-platform round-trip); bridge `presetTree`/`savePreset`/`applyPreset`/`addPresetGroup`/`rename`/`delete`/`move`/`import`/`export`; new `PresetBrowserSheet` UI; the session-only presets array was retired and mutations funnel through the view model (Edit ▸ Effect Presets…, ⇧⌘P). **Original spike plan, for reference:** Relocate `EffectPresetManager` to wx-free `src-core`; the wx UI (`EffectTreeDialog`) and the desktop apply path (`EffectsGrid::Paste`) **stay desktop-only** — both platforms keep their own apply path over the shared serialized settings string. Persist to **`<showFolder>/xlights_effectpresets.json`** (+`.jbkp`) — the desktop format, so presets **round-trip cross-platform**. iPad: own an `EffectPresetManager` in `iPadRenderContext`, retire the session-only `SequencerViewModel.presets` array, and add bridge methods `presetTree()` / `findPreset(byPath:)` / `savePreset(fromSelection…)` (reuse the existing copy/paste serializer for the `\tLAYER:n` format) / `applyPreset(atPath:toRow:atStartMS:)` (route through existing `addEffect`/`replaceEffectSettings`) / add-group/rename/delete/move / import-export. Acceptance gate: `check_core_include_boundaries.sh` + a multi-layer preset round-tripping desktop↔iPad. This unlocks PRE-2/3/4 below.

## Current parity

- **Section 23** — 28 features: 2 implemented, 5 partial, 20 missing, 1 out-of-scope. The lowest-parity area in the whole app. The two implemented rows are the persistence layer (`<Jukebox>` node round-trips via shared core) and the touch Effect Dropper (`EffectPaletteView`). Everything user-visible in presets and *all* of jukebox is partial or missing.
- **Section 4** — 34 features: 10 implemented, 7 partial, 7 missing, 10 out-of-scope. The implemented core (timeline zoom, Display Elements, the four inspector tabs, preview/inspector toggles + detach-to-scene) is the platform we build on. The 10 out-of-scope rows are all the wxAUI perspective/dock subsystem. The portable remainder is small: AC ramps toggle, Dock All, Reset-to-Defaults, AC Lights paint, and the dockable-utility-panel rows that overlap directly with the Section 23 preset/jukebox work.

## Approach

The defining discovery of this theme: **the preset tree data model already exists and is almost portable.** `src-ui-wx/effects/EffectPresetManager.{h,cpp}` defines `EffectPresetItem` / `EffectPreset` / `EffectPresetGroup` / `EffectPresetManager` — a full pugixml + nlohmann/json tree with add/remove/move/rename, `FindPresetByPath`, `ImportFromXml`, `FixRgbEffects` name-fixup, `Load(effectsNode)` / `Save(visitor)` against the rgbeffects `<effects>` node, and `LoadJsonFile`/`SaveJsonFile`. It is header-clean (pugixml/json only); the `.cpp` has just three shallow wx usages (`wxFileName::GetForbiddenChars`, two `wxString::Format` calls in name-fixup) plus `shared/utils/wxUtilities.h`. So the core work is **relocate `EffectPresetManager` to `src-core/effects/`**, swap those three wx calls for std equivalents, register it in the desktop project files, and the *exact same* class then drives both desktop and iPad. This is a parity win for the desktop too (the class leaves the wx UI layer).

From there the iPad side is pure presentation + bridge wrapping, following the established `savedPalettes` / `savedValueCurves` template on `XLSequenceDocument`:

- **Bridge (`src-iPad/Bridge/`)** — `XLSequenceDocument` gains a flat, Swift-friendly facade over the core manager: enumerate the tree (path + isGroup + layer-count/duration metadata), add/rename/move/delete preset and group, capture the current multi-effect selection into a preset (reusing the iPad's existing `clipboardEntries`/`copySelectedEffects` data — the same serialized form desktop's `GetSelectedEffectsData` produces, with `\tLAYER:n` markers), apply a preset by path (reusing the existing `pasteAtRow` multi-row/layer paste path), update-from-selection, and import/export `.xpreset`. The tree itself is owned by `iPadRenderContext` (mirroring `xLightsFrame::_effectPresetManager`) and loaded/saved with the rgbeffects file the iPad already reads. All mutations funnel through `SequencerViewModel` so undo + row reload stay in one place. `NS_SWIFT_NAME(...)` per the header convention.
- **Jukebox** — shared core `JukeboxButtonData` / `JukeboxButtonMap` (`src-core/render/JukeboxButtonData.h`, `Load/SaveJukeboxButtons`) already round-trips the `<Jukebox>` node; only UI + bridge are missing. Bridge exposes read/write of the button map and the cascading Model→Layer→Time lookups (analogous to desktop `GetAllElementNamesWithEffects` over `SequenceElements`, already in `iPadRenderContext`). Playback reuses the iPad's existing loop-region transport (`audioPlaySegment`, `toggleOutput`/`sendOutputFrame`).
- **Touch idioms** — presets and jukebox become SwiftUI sheets/inspector panels with `.searchable`, `List` + `.onMove`, swipe-to-delete, and long-press context menus rather than dockable AUI panes. Preset apply hangs off the existing grid long-press menu (already wired) plus a new browser sheet. Animated GIF previews become SwiftUI-native animated thumbnails rendered through the already-present `iPadRenderContext` render engine (no on-disk GIF).

## Quick wins (Rollup C — cheap, high-leverage, do early)

- **VIEW-2 — Show AC Ramps toggle (S).** Core `DrawEffectBackground` already takes the `drawRamps` bool and the iPad passes it; only the hardcoded `true` at `EffectsMetalGridView.swift:577` plus a `@AppStorage` flag and a View-menu item are missing. Trivial, immediately user-visible.
- **VIEW-3 — Dock All / Reset to Defaults (S).** Each detached scene already has an individual Dock command; the aggregate is a loop. Reset is clearing two `@AppStorage` keys + re-showing/docking. Both are tiny convenience wins on the existing window plumbing.
- **VIEW-1 — Effect Dropper icons (S).** Bridge already supplies per-effect BGRA icons (`iconBGRAForEffectNamed`, used by the grid); swapping the palette's text labels for icon buttons is a contained view change that closes a visible "partial" row.
- **PRE-6 — Apply mode toggle (S).** Becomes a near-free add the moment PRE-3 lands the `\tLAYER:` capture/apply (auto-detect from marker presence).
- **VIEW-7 — Select Effects panel (S).** The placed-effect enumeration over `SequenceElements` is already bridged; this is a SwiftUI filter form (effect-type / buffer-style / time / color + sensitivity) feeding the existing multi-select highlight path. Contained, and a high-leverage navigation/selection aid distinct from SEQ-3's global Select-All.

These five bundle to roughly **2 days** and clear three "partial" rows plus two P2s with minimal risk, while PRE-1/PRE-2 (the model relocation) proceed in parallel.

## Out of scope for this theme

These rows are out-of-scope; full reasons live in the master out-of-scope doc (Rollup B). One line each:

- **Reset Toolbars** — no wxAUI toolbars on iPad; fixed SwiftUI layout has no toolbar state to reset.
- **Perspectives: Save Current / Save As New / Edit-Load / Auto Save / dynamic menu / shortcut item** — all serialize wxAUI pane-geometry strings; iPad has no AUI docking, so nothing to capture/manage. Pane sizes + detach-state already auto-persist via `@AppStorage`.
- **Perspective layout engine (wxAUI dock/float/persist)** — scene-based iPadOS windowing (Stage Manager / Split View) is intentionally substituted for free-form docking; a SwiftUI AUI clone is an enormous effort against the platform grain.
- **Suppress Dock: House Preview / Model Preview** — force-floating a pane; no dock-vs-float state machine exists on iPad (detach-to-scene is the equivalent and is per-instance user-controlled).
- **Jukebox tooltip/state export for xSchedule** — a wx-label IPC contract for the desktop-only, separate-repo xSchedule app; no consumer on iOS.

Note: several "Windows: …toggle" rows in Section 4 (Effect Presets, Effect Dropper, Jukebox) are *not* listed here — they overlap directly with the in-scope PRE-*/JUK-*/VIEW-1 items above and are delivered there.

## Risks / open questions

- **PRE-1 relocation regressions.** Moving `EffectPresetManager` to `src-core/` touches the desktop preset path (`EffectTreeDialog::InitItems`, `tabSequencer.cpp` update/apply). Must build and smoke-test the desktop preset tree (save/apply/import) after the move, not just the iPad lib. The `check_core_include_boundaries.sh` gate will catch any residual wx include — that's the acceptance signal that the relocation is clean.
- **Preset format compatibility.** iPad presets must serialize byte-compatibly with desktop's `GetSelectedEffectsData` (`\tLAYER:n` markers, version stamps) so presets created on either platform apply correctly on the other. Validate by round-tripping a multi-layer preset desktop→iPad and back.
- **Apply-at-drop-location on touch.** Desktop applies presets at a paste/drop point; iPad currently applies to the *selected* effect(s). PRE-3 should define the touch gesture for "apply here" (long-press-on-empty-cell vs a placement mode) — confirm the intended idiom with design before building PRE-4's browser apply path.
- **AC-1 sizing.** AC fill logic is in `src-ui-wx`, not `src-core`. The core extraction could be larger than the "L" estimate suggests; if so, demote AC-1 to Phase 3 or split into "core extract" then "touch UI" sub-tasks. It is the only XL/uncertain item in the theme.
- **rgbeffects ownership on iPad.** Confirm `iPadRenderContext` already loads/holds the rgbeffects document at the right lifecycle point to host the preset manager (it owns the equivalent of `xLightsFrame` state) — PRE-2 assumes the `<effects>` node is reachable and re-saved with the file.
- **GIF vs animated thumbnail.** PRE-8 deliberately diverges from desktop (no on-disk GIF). Confirm there's no cross-platform expectation that preset preview GIFs are shared files; if presets are expected to carry a preview image across platforms, revisit.
