# iPad Parity Plan — AI Features & Automation / Scripting

**Scope:** Gap-analysis Area 21 (AI Features) and Area 22 (Automation, Scripting & REST API). The unifying theme: most of the heavy lifting already lives in shared `src-core/` (the `ai/ServiceManager` + providers; the operations behind every automation command). What's missing is the *surface* — SwiftUI editing affordances on top of the AI flows, and an in-process command-dispatch layer on top of `XLSequenceDocument`. The dead transport (CLI/argv, HTTP listener port, Python, xlDo) is explicitly out of scope.

> **Status (2026-06-01) — landed (build-verified, `xLights-iPad` app):**
> - ✅ **AUTO-3** — "Speech to Lyrics…" entry in the timing row-header context menu (gated on audio loaded + a Speech-to-Text service via `canSpeechToLyrics(rowIndex:)`), routing through the existing `generateAILyricTrack` by opening the unified Add-Timing sheet pre-selected on AI Lyrics (`pendingSpeechToLyricsRowIndex`).
> - ✅ **AI-1** — AI image style presets. New `XLAIImageSession.imageProperties()` / `setStringProperty:value:` expose the generator's `GetProperties()` / `SetProperty()` via a shared `XLAIServiceProperty+Internal.h` converter; `AIImageGenerationSheet` renders Choice properties (e.g. Style) as Pickers, refreshed on service change.

---

## Current parity

- **Area 21 (AI), 26 features:** 14 implemented, 3 partial, 8 missing, 1 out-of-scope. The provider/service stack, Keychain secrets, Services config + Test panel, AI palette generation, embedded AI image-into-Pictures, and on-device speech-to-text lyric tracks all work. Gaps are concentrated in the AI Image post-generation editor (style presets, crop, resize, reset, black-background flood-fill) and the AI model-mapping-on-import feature.
- **Area 22 (Automation), 58 features:** 0 implemented, 28 partial, 15 missing, 15 out-of-scope. "Partial" here almost always means *the operation exists as a wired `XLSequenceDocument` bridge method but there is no command-dispatch wrapper to invoke it by name*. The single keystone gap is a wx-free JSON command dispatcher (the iPad analog of `xLightsFrame::ProcessxlDoAutomation`); building it converts ~25 partial rows into working commands and would also be the foundation for a (deferred, high-risk) Lua engine.

---

## Approach

1. **AI editing gaps are mostly pure SwiftUI/client-side** (UIImage/CGImage), with two exceptions that need shared-core exposure: image-generator *style properties* (already in core; just unwired) and a `RemoveBlackBackground` flood-fill (currently a wx-only helper — port to a wx-free pixel routine, ideally in `src-core/`, so the desktop can share it back). Touch-native crop (drag-rectangle on the Metal/`Image` preview) is a *better* fit for iPad than the desktop marquee.
2. **AI model mapping reuses the shared `claude::GenerateModelMapping` engine** (`src-core/ai/claude.cpp:251`, `aiBase::MappingModelInfo` at `aiBase.h:134`). The only net-new code is a wx-free builder that fills `MappingModelInfo` from the iPad's model managers (the desktop builder `DoStructuredAIMapping` is ~110 lines of wx), a bridge method on `XLImportSession`, and an "AI Map" button in `ImportEffectsView`.
3. **Automation = one dispatcher, many thin wrappers.** Build a wx-free `XLCommandDispatcher` bridge that takes a command name + JSON params (`nlohmann/json` is already a core dep) and routes to the *existing* `XLSequenceDocument` / `XLImportSession` bridge methods. Each "Command: X" partial row becomes a `case` in that dispatcher plus a small JSON in/out marshalling shim — almost no new core work. The dispatcher is the gate for everything downstream (Lua `RunCommand`, a future App Intents surface).
4. **Lua is a DEFERRED, high-risk scripting path** (`liblua.a` already ships in `lib-ios/`). It is strictly downstream of the dispatcher (`RunCommand` is a one-line forward), but it is **not** an MVP item: it is viable **only for bundled or in-app-authored `.lua` scripts** that ship inside the signed app or that the user types/edits on-device — **never** scripts downloaded from a remote source. The classic "Lua JIT is App-Store-hostile" objection is *moot* here because iPad Lua runs the standard interpreter (bytecode is interpreted data, not JIT-compiled native code), but two real concerns remain: the user-script-API surface still has to be mapped onto the dispatcher, and shipping any scripting engine carries App Store review risk that must be evaluated before committing. Python and the HTTP listener stay out. Downloading community scripts (old AUTO-11) is out of scope — App Store guideline 2.5.2 forbids downloading/executing code (see `99`).
5. **Prefer iOS idioms over literal ports:** sheets and `.searchable`, long-press menus, `UIApplication.open(URL)` for "open controller in browser", Files-app share/QuickLook instead of "reveal in Finder", and (future) App Intents/Shortcuts instead of a localhost REST port.

## Out of scope for this theme

(One line each; the master out-of-scope rollup has the authoritative list and reasons.)

- **AI plugin DLL loading** — iOS App Store forbids loading executable code outside the signed bundle; `ServiceManager` is intentionally built with an empty plugin dir.
- **HTTP/REST automation server (A/B port)** — marginal value on a sandboxed single-window touch app; *technically* possible via Network.framework, so flagged low-priority rather than hard-blocked. Natural replacement is App Intents/Shortcuts (unbuilt).
- **Automation port preference (Disabled / Port A / Port B)** — only configures the nonexistent A/B listener; no argv on iPad.
- **`exportVideoPreview`** — needs an FFmpeg/AVAssetWriter encode pipeline; a net-new video subsystem, not an automation gap.
- **`playJukebox` / `jukeboxButtonTooltips` / `jukeboxButtonEffectPresent`** — no Jukebox feature on iPad to drive (see Jukebox plan).
- **`closexLights`** — iOS apps cannot programmatically `exit()`; OS owns lifecycle.
- **Python scripting engine (pybind11 / CPython)** — App Store 2.5.2 + no Python lib in iOS deps; (deferred, high-risk) Lua is the only candidate scripting path, and only for bundled / in-app-authored scripts.
- **Run Scripts: GitHub / community-script download (was AUTO-11)** — downloading and executing remote `.lua` violates App Store guideline 2.5.2 (no downloading code). Hard-blocked, not merely low-priority; authoritative entry is in `99` §1. The same rule is *why* the deferred Lua engine (AUTO-7/8) is restricted to bundled or on-device-authored scripts only.
- **Run Scripts: View Script / Open Folder context menu** — no "reveal in Finder" / external editor on iOS (QuickLook/share is the closest idiom).
- **xlDo CLI + template `%P1..%P9` + `-s` script gen + `startxLights` launcher** — standalone CLI binary presupposing a shell + localhost automation port; neither exists on iOS.
- **App launch flags: headless `-r`/`-cs`, `-a/-b`/`-o`/`-s`/`-m`/`-w`** — no argv on iOS; the underlying ops exist as interactive UI (Batch Render, Check Sequence, FolderConfig, output toggle).
- **Internal automation request helper (`xLightsRequest`)** — an HTTP self-call to the local port; iPad drives the core directly via the bridge, so the round-trip is pointless.

## Risks / open questions

1. **Stable effect IDs (AUTO-2).** The desktop addresses effects by model+layer+effectID; the iPad bridge addresses by (row, index). A correct ID↔index resolver is the trickiest part of scripted effect editing — effects *do* carry stable IDs in core, but the bridge must expose and round-trip them. Get this contract right before building AUTO-5's `getEffectIDs`.
2. **`highdef` render override (AUTO-4).** `iPadRenderContext` hard-codes `IsLowDefinitionRender()==true` as a deliberate perf choice. Honoring `highdef=true` from `renderAll`/`batchRender` needs a temporary, scoped override — evaluate whether it's worth the complexity or whether the command should simply document that iPad always renders low-def.
3. **`RemoveBlackBackground` location (AI-2).** Porting it to `src-core/` lets the desktop adopt the shared routine (per the parity guidance), but the desktop version is a wx pixel helper — confirm a wx-free pixel API exists or add one in `src-core/utils/`. If that's too invasive, a `.mm`-local CGImage routine is acceptable but loses the desktop share-back.
4. **`MappingModelInfo` builder fidelity (AI-4).** The structured mapper's quality depends on faithfully reproducing the desktop's type / node-count / alias / submodel / group-member extraction. Validate against the same fixtures the desktop uses; a partial builder will silently produce worse mappings.
5. **Synchronous prompts from a background Lua thread (AUTO-9).** Bridging a blocking script thread to main-actor SwiftUI alerts is the known-fiddly part of the Lua surface. Decide early whether scripts run on a background thread (needs the bridge) or are constrained to non-interactive commands in v1. (Downstream of the deferred AUTO-7; do not build before the Lua engine ships.)
6. **Dispatcher error contract (AUTO-1).** The desktop returns HTTP-ish status codes (e.g. 504 on unsaved-close). Define the iPad JSON error shape once, up front, so every command and the eventual Lua `RunCommand` table conversion are consistent.
7. **Command surface drift.** As `XLSequenceDocument` grows, the dispatcher's command map must track it. Consider a single table mapping command-name → bridge selector so new commands are additive and discoverable, rather than a sprawling `if/else` chain.
8. **Lua engine deferral + App Store review risk (AUTO-7/8).** Lua is DEFERRED, not MVP. The legacy "Lua-via-JIT is App-Store-hostile" framing (from `README` / the prior `future-help-diagnostics.md` Run-Scripts hard-miss note) is **superseded and now inaccurate**: iPad Lua runs the standard *interpreter*, so the JIT objection does not apply. The two live blockers are instead (a) re-mapping the user-script API onto the dispatcher (desktop scripts call a UI-automation surface that does not exist verbatim on iPad), and (b) App Store review risk for shipping *any* embedded scripting engine — which is why scope is restricted to bundled / in-app-authored scripts and remote-download (old AUTO-11) is hard-blocked under guideline 2.5.2. Make the review-risk go/no-go call before committing engineering to AUTO-7. If any prior README/help text still says "no plans for Lua" or cites the JIT objection, treat it as stale and defer to this entry.
