## Current parity

Area 15 has **39 features: 22 implemented, 5 partial, 9 missing, 3 out-of-scope.** The high-value core is already at parity: the 8-slot palette grid, per-swatch native `ColorPicker`, the full ColorCurve editor (gradient + 9 spatial/temporal modes), and the complete Value Curve editor (all 23 types, custom points, reverse/flip/wrap, time offset, music/timing binding, `.xvc` presets, clipboard) all round-trip byte-for-byte with desktop through the shared `XLValueCurve`/`XLColorCurve` wrappers. Brightness/Contrast/HSV-adjust, Chroma Key, and Sparkles (incl. music + color + value-curve) ride on the shared `Color.json` metadata. The work that remains is almost entirely SwiftUI affordances and a few thin bridge wrappers over already-shipping `src-core` ops — no new color math.

## Approach

- **Shared core already does the work.** Palette serialization (`C_BUTTON_PaletteN` / `C_CHECKBOX_PaletteN` keys), used-colour enumeration, and colour replacement all live in `src-core` and are linked by the iPad lib today. Specifically `SequenceElements::GetUsedColours(bool)` and `SequenceElements::ReplaceColours(RenderContext*, from, to, bool)` (`src-core/render/SequenceElements.cpp:1536/1556`, with the recursive `EffectLayer`/`Effect` impls) need **only** new `XLSequenceDocument` wrappers — no algorithm port.
- **Palette toolbar ops are pure Swift.** Reverse / shift-left / shift-right are slot rotations over the eight `C_BUTTON_`/`C_CHECKBOX_` key pairs the iPad already reads in `ColorPaletteView.swift`; they can run entirely in the view-model with no bridge call, mirroring `ColorPanel::OnBitmapButton_ReverseColoursClick` / `ShiftColours{Left,Right}` (`src-ui-wx/color/ColorPanel.cpp:1240/1257/1285`).
- **Bulk-edit reuses an existing funnel.** `SequencerViewModel.applyValueToAllSelected(_:forKey:)` (`SequencerViewModel.swift:2665`) already writes one key to every selected effect as a single undo step via `setEffectSettingValue`. The per-parameter value-curve bulk-edit gap is just wiring the VC button's long-press menu to call this with the `*_VALUECURVE_*` key — and likewise a palette-scoped "Update Palette" is `applyValueToAllSelected` looped over the 16 `C_` palette keys only.
- **Native touch idioms over desktop chrome.** No owner-drawn dropdowns or docked toolbars: use a SwiftUI palette-action `Menu`/toolbar row, a long-press menu on the VC button, `.dropDestination` for drag-onto-swatch, and a sheet for colour-replace / the dropper panel. The native `ColorPicker` stays the canonical color editor (the custom HSV dialog is out-of-scope by design).
- **All mutating ops funnel through `SequencerViewModel`** so undo registration + row reload happen in one place, consistent with the existing bulk paths.

## Quick wins (Rollup C / close-to-done)

- **`COL-2` (per-parameter VC bulk-edit)** — the only Rollup-C item in this theme. It is a single long-press menu on `ValueCurveButton` wired to the *already-implemented* `applyValueToAllSelected(_:forKey:)`. Effort S, no bridge work, closes a partial that shows up in both Area 14 and Area 15.
- **`COL-3` (reverse/shift palette)** and **`COL-1` (Update Palette)** are the next cheapest: both are pure Swift over keys the palette view already reads/writes, no new core or bridge surface. Ship these three first for visible parity wins before the bridge-touching COL-5/6/7 cluster.

## Out of scope for this theme

(Reasons summarized; the master out-of-scope rollup — `_raw-gap-analysis.md` Rollup B — has the full text.)

- **Custom HSV/RGB color picker dialog** — iOS ships a rich native `ColorPicker` (HSB grid, sliders, hex, eyedropper); a bespoke HSV-disc dialog fights the platform and adds no value.
- **Reset panel when changing effects** — a desktop `wxConfig`-backed preference for the persistent docked ColorPanel; the iPad inspector rebuilds per-selection, so the behavior has nothing to govern.
- **Palette display size preference (Normal/Large)** — exists only to manage the desktop's fixed-width docked panel; iPad palette rows are full-width, touch-sized, and adapt automatically.

## Risks / open questions

- **`ReplaceColours` undo integration (COL-5).** Core `ReplaceColours` takes a `RenderContext*` and a `UndoManager&` and registers its own undo through the *core* `UndoManager`. The iPad sequencer uses SwiftUI's Foundation `UndoManager`, not the core one (see Area 2 Undo partial). Decide whether the colour-replace bridge wraps the whole op in one Swift-side undo closure (snapshot affected effects before/after) or whether the core undo is surfaced — the simplest is a Swift-side compound undo around a single bridge call, consistent with the other bulk paths.
- **Render invalidation scope after replace.** A sequence-wide replace can touch many rows; the bridge should return the set of affected rows (or the iPad should re-render dirtied rows) rather than forcing a full re-render. Confirm `ReplaceColours` marks effects dirty in a way the iPad render-tracking observes.
- **ColorCurve payload in COL-5/COL-7.** Confirm the replace "to" value and drag payload accept a serialized ColorCurve blob (desktop's `ColourReplaceDialog` allows replacing with a color curve via `ColorCurveButton`), and that `ReplaceColours` matching keys on flat-vs-curve behaves as on desktop.
- **`.scss` palette format (COL-4).** The analysis flags `.scss` palette parsing as unconfirmed in `PaletteSerializer`; verify before claiming dropdown parity, otherwise document `.xpalette`-only support.
- **Curve drop rejection on flat-only wells (COL-7).** Ensure chroma/sparkle color wells reject ColorCurve drops exactly as desktop does, so an invalid curve blob can't be written to a flat-only `C_COLOURPICKERCTRL_*` key.
