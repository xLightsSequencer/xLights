**Scope:** Gap-analysis Section 14 (*Effects: All 55 Effect Panels & Rendering*). This theme is about per-effect **panel** behavior, not rendering — every effect already renders correctly through the shared core. The work is reproducing the desktop's code-driven dynamic-UI logic (`ValidateWindow()` enable/disable + dynamic labels) and a couple of un-populated dynamic-option choices that the JSON metadata + iPad rule engine do not yet capture.

## Current parity

Section 14 is the strongest area in the whole gap analysis: **70 features — 55 implemented, 14 partial, 1 missing, 0 out-of-scope.** The JSON-driven panel framework, the visibility-rules engine, value curves, the cross-cutting Color/Buffer/Blending panels, and all 55 renders are ported or shared (every effect renders via the same `src-core` `RenderableEffect`/`RenderEngine` through `iPadRenderContext`, and `effectmetadata/*.json` is loaded onto each `RenderableEffect` by the shared `EffectManager`). The 14 partial rows are **panel-level dynamic UI** the desktop implements in C++ `ValidateWindow()` and which is absent from the JSON `visibilityRules` — so on iPad those controls are always visible/enabled, letting users set parameters the render then ignores. There are also two un-wired dynamic-option choices (audio-track lists, Text lyric track) and a long tail of interactive-canvas / convenience gaps: the three desktop Effect Assist sub-panels (EA-1 Sketch — mostly shipped; EA-2 Pictures pixel-paint; EA-3 Morph 4-corner quad), the animated-GIF per-frame **timing** editor (G6, distinct from paint and from preview), Moving Head multi-tab DMX canvases, animated preview thumbnails, Shader download, the full RemapDMXChannels grid, and Shader dynamic-uniform grouping.

## Approach

Three layered strategies, in priority order:

1. **Author the missing dynamic UI into the shared JSON (`resources/effectmetadata/*.json`), not into platform code.** The iPad rule engine (`EffectSettingsView.swift:721-803`, schema in `EffectMetadata.swift:154-182`) already evaluates `equals / notEquals / oneOf / notOneOf / greaterThan / startsWith / any` with `enable / disable / show / hide` clauses — this is *exactly* the shape of the desktop `IsType(type, {...})` enable matrices in `VUMeterPanel::ValidateWindow()` and the source/direction enables in `TextPanel::ValidateWindow()`. Crucially the desktop `VUMeterPanel` / `TextPanel` are themselves `JsonEffectPanel` subclasses; their `ValidateWindow()` runs *in addition to* the JSON rules. So authoring `visibilityRules` into the shared JSON closes the iPad gap **and** lets the desktop subclass be slimmed later (optional follow-up, out of this theme's critical path). This is data-only work — no new core algorithm, no bridge method.

2. **Extend the rule schema only where enable/disable can't express it.** The one case JSON rules cannot represent today is the Adjust panel's *dynamic label swap* (Value 1 → "Adjust by:" / "Minimum:" / "Shift by:" per Action). That needs a small additive metadata field (a `labelRules` / `dynamicLabel` map keyed off another property), parsed in `EffectMetadata.swift` and applied in `EffectPropertyView`. It is shared (helps desktop too).

3. **Thin bridge wrappers for un-populated dynamic-option choices.** `SequencerViewModel.dynamicOptions(source:propertyId:)` (`SequencerViewModel.swift:2511`) is missing an `audioTracks` case; the bridge already exposes the data (`XLSequenceDocument.altTrackCount` / `altTrackDisplayName(at:)` at `XLSequenceDocument.h:1657`). Adding the case + setting `dynamicOptions` on the relevant JSON choices is a few lines each.

Interactive-canvas gaps (Pictures paint, Moving Head) follow the established touch-idiom pattern already used for Morph/Sketch (inline custom rows), but are P2/P3 and explicitly the long tail of this theme.

## Out of scope for this theme

This theme has **no out-of-scope rows** — Section 14 lists 0 out-of-scope features (every effect renders on iPad). Adjacent out-of-scope items belong to other themes and are tracked in the master out-of-scope doc:

- *Effect Assist Window mode* (Preferences area) — iPad has no docked Assist pane; inline custom rows replace it, so an Always-On/Off/Auto mode has nothing to govern.
- *Custom HSV/RGB color picker dialog* (Color/Palettes area) — iOS native `ColorPicker` is the platform idiom.
- *Incompatible-video transcode on import* (Media area) — no FFmpeg encode in the iPad lib; relevant to the Video effect picker, handled as warn-and-skip (constrains `FX-11`'s Video preview, not this theme's core work).

## Risks / open questions

- **Schema extension scope (`FX-4`/`FX-7`).** Adding a `dynamicLabel` field touches the shared schema (`resources/effectmetadata/_schema.json`) and the desktop `JsonEffectPanel` parser as well as `EffectMetadata.swift`. Keep it additive and optional so desktop ignores it until wired; confirm the desktop JSON validator (CI) accepts the new key. Build the iPad lib (`xLights-iPadLib`) **and** the desktop after any schema/parser change.
- **`oneOf` value fidelity (`FX-1`).** The VU Meter Type enum has ~60 string values; the rules must match the exact option strings in `VUMeter.json` (e.g. "Timing Event Timed Sweep 2"). A single typo silently disables the wrong control. Generate the rules programmatically from the desktop `IsType` lists and the JSON `options` array, then spot-check the greyed-out state for several representative types.
- **Desktop double-application.** Because desktop `VUMeterPanel`/`TextPanel`/`AdjustPanel` keep their `ValidateWindow()`, adding JSON rules makes the rule run on *both* platforms — verify the JSON rules agree with the C++ so the desktop doesn't flicker or fight itself. Where they agree exactly, file a follow-up to delete the redundant C++ (not required for parity).
- **"Text has a value" predicate (`FX-2`).** The desktop source-priority test is `!text.IsEmpty()`, which the current operators (`equals`/`notEquals`/`startsWith`) can't express directly for arbitrary content. Decide whether to add a `nonEmpty` operator to the rule engine or accept a minor behavioral divergence; lean toward the small `nonEmpty` operator since it's broadly useful.
- **Moving Head (`FX-9`) sizing.** XL and largely independent of the rest of the theme; could be split into its own mini-plan if it threatens Phase-2 timelines. The four canvases are independent and can land incrementally.
