# iPad Parity Plan — Preferences & Settings

**Scope:** Gap-analysis Section 20 (Preferences & Settings, 77 prefs). Also folds in the cross-cutting preference rows surfaced in adjacent sections that are really "global toggles over already-wired behavior": Snap to Timing (§11), Paste As mode (§2 Rollup C), Check-Sequence suppression (§3/§17), and the autosave-interval persistence caveat (§1). Goal: design **one consolidated iPad Settings surface** and map every desktop pref to **port / re-conceive / drop**.

**Desktop surface for reference:** the `wxPreferencesEditor` has **10 panels** totalling roughly **96 options**: Folder/Sequence-File, Effects-Grid (11), View (12), Other (18), Backup (6), Random-Effects (~80 per-effect toggles, counted as one feature here), Network/Output, AI Services, and the per-page Render knobs. Of these, ~30 land in-scope across PREF-1…PREF-29 below and ~40 are deliberately dropped (see "Out of scope" and `99` §7) so the in-scope-vs-out accounting is explicit and auditable.

## Current parity

Per the Summary table, Section 20 has **77** prefs: **9 implemented, 12 partial, 40 missing, 16 out-of-scope**. The implemented core is the right one — folders (show/media/fseq/base-show with bookmarks), FSEQ-on-save, AI Services config + test, and the per-sequence Render-tab knobs (frame rate, model blending, autosave interval). (The desktop **FSEQ Version** choice — `V1`, `V2 ZSTD`, `V2 Uncompressed`, `V2 ZLIB`, `V2 ZSTD/sparse (Default)`, `SequenceFileSettingsPanel.cpp:107-112` — is owned by FSEQ-1 in doc 09; the iPad currently hardcodes the default sparse-ZSTD variant.) The 40 "missing" are overwhelmingly **low-value polish toggles**, not algorithms; the genuinely high-value gaps are a small set: **Render on Save, Paste As mode, Effect Grid row-height, Snap-to-Timing disable, and Check-Sequence suppression** — every one of which is a SwiftUI toggle plus a thin bridge wrapper over an already-shared core call (verified below). The big structural gap is that there is **no single Preferences entry point**: knobs are scattered across `FolderConfigView`, `SequenceSettingsSheet` Render tab, and `AIServicesSettingsSheet`.

## Approach

- **Build the shell first, migrate into it.** Create one `SettingsView` (a `NavigationStack` / sectioned `List` presented as a sheet from a `Settings…` toolbar/menu item, Cmd-, on hardware keyboard). It hosts app-global prefs and *links out* to the existing `FolderConfigView` and `AIServicesSettingsSheet` rather than re-implementing them. Per-sequence settings (frame rate, model blending, autosave) stay in `SequenceSettingsSheet` — they are document properties, not app prefs — but we add cross-links so the surface is discoverable.
- **App-global toggles are `@AppStorage`, surfaced once, read at the point of use.** Most missing rows are a single `@AppStorage` Bool/enum plus gating an existing code path. We centralize their *declaration* in a small `AppSettings` namespace (mirroring `FolderConfig`) so views read one source of truth.
- **The core is already shared and the setters already exist.** Confirmed shared-core APIs that the bridge only needs to *wrap* (no new algorithms):
  - **Render Cache mode/size:** `RenderCache::Enable("Disabled"|"Locked Only"|"Enabled")` and `SetMaximumSizeMB(size_t)` in `src-core/render/RenderCache.h`.
  - **Check-Sequence suppression:** `SequenceCheckerCallbacks::IsCheckOptionDisabled(option)` and `RenderCacheModeOverride()` hooks **already exist** in `src-core/diagnostics/SequenceChecker.h` — `iPadRenderContext` just needs to return per-check Bools from `@AppStorage`.
  - **E1.31 sync / Force Local IP:** `OutputManager::SetSyncEnabled(bool)` / `IsSyncEnabled()` and `SetGlobalForceLocalIP(std::string)` / `GetGlobalForceLocalIP()`; local-IP enumeration via `ip_utils::GetLocalIPs()` (all `src-core/outputs/OutputManager.h`, `src-core/utils/ip_utils.h`).
  - **Effect-grid row height:** `GridMetrics.rowHeight` already carries a comment that it exists "so a future preference pane can adjust them"; the Metal grids already read `GridMetrics`. Pure plumbing.
  - **Snap / fade-hints / AC-ramps / effect-backgrounds:** the draw/snap logic is already isolated (`EffectsMetalGridView` `snapMS`, `drawRamps`, fade-bar drawing, `effectBackgroundAccumulator`); each just needs a Bool gate.
- **Re-conceive, don't port, where the idiom differs.** Backup → iCloud-aware per-show ZIP backup action, not F10/F11 + purge-interval machinery. Icon size / handle size / palette size → already auto-sized for touch; expose only on user demand. Color Manager editing → defer (color store lives in wx, not core).
- **Apply-immediately convention.** App-global toggles apply immediately (matches macOS). Folder changes keep their explicit Cancel/Done because they trigger a show reload. Document this once in the new `SettingsView` so the §20 "no unified apply policy" caveat is resolved.

## Out of scope for this theme

(The master out-of-scope list — Rollup B — has the full reasons; one-liners here so this plan stands alone.)

- **Effect Assist Window mode** — no Effect Assist pane on iPad; inline inspector rows replace it, nothing to govern.
- **Auto Show House Preview** — preview is an embedded/detached scene; no floating auto-appear window to toggle.
- **Disable key acceleration / Zoom To Cursor** — touch-first; key-repeat and cursor-anchored zoom are OS-managed (pinch centroid is the equivalent).
- **Low Definition Render** — `iPadRenderContext::IsLowDefinitionRender()` hard-coded true for 4 GB-device safety; a toggle risks OOM.
- **xFade/xSchedule sync port** — desktop companion apps over shared-memory IPC; no counterpart on iOS.
- **Suppress Dark Mode (Windows only)** — Windows-only; iPadOS follows system appearance per HIG.
- **Hardware Video Decoding / Renderer / GPU Rendering / Shaders on Background Threads** — Apple uses VideoToolbox automatically; the render context is already device-tuned; ShaderEffect is single-threaded on Metal by design.
- **Video Export Codec / Bitrate** — no on-device video-export pipeline (would need a future AVAssetWriter feature).
- **Tip Of The Day min level / recycle** — no tips engine on iPad.
- **Use custom color picker (experimental)** — iPad uses the native SwiftUI `ColorPicker`; no wx custom picker to switch to.

## Risks / open questions

- **Render on Save (PREF-2)** quietly produces a **low-def** fseq because `iPadRenderContext` is hard-coded low-def. Make sure the toggle's help text says so, or users may assume save-time render matches desktop fidelity.
- **Render Cache mode "Disabled" + big sequences (PREF-9)** changes memory/CPU behavior on 4 GB devices — validate before defaulting away from Enabled; keep Enabled as the default.
- **Paste As mode (PREF-3)** must round-trip identically to desktop's `xLightsPasteAsLayers` so cross-platform copy/paste behaves consistently; verify the relative-vs-layers branch against a multi-layer clipboard.
- **`@AppStorage` vs core persistence:** app-global toggles live in `UserDefaults`, but several map to desktop prefs stored in `xlights_rgbeffects.xml` / OtherSettings (e.g. ForceLocalIP, sync). Decide per-pref whether iPad persists locally only or writes back through the bridge to the show file — output prefs (PREF-11) arguably belong in the show file for consistency with desktop; grid-appearance prefs are fine as device-local `@AppStorage`.
- **Color Manager editing** is deliberately *not* in this plan (color store lives in wx, not shared core; §15 row is L-effort and low value). Flag if users actually want grid-color theming.
- **Where does the Settings sheet live** — a single in-app `SettingsView` sheet vs an iOS `Settings.app` bundle. Recommend in-app sheet (apply-immediately, can link to folder pickers and run live reloads); a Settings.app bundle can't trigger reloads or enumerate local IPs.
- **Controller ping interval** — the desktop `Controller ping interval in seconds (0=Off)` pref (`OtherSettingsPanel.cpp:278`, `SetControllerPingInterval`) is **not** a preferences item here: it belongs with the controller online/ping work in **doc 07** (see CTL-8 / CTL-23). Listed here only so the Other-panel option count is complete and the pref isn't double-owned.
- **Custom KeyBindings editor framing** — PREF-29 is kept **in-scope as P3**. The prior `README`/`future-help-diagnostics` "hard miss" framing for a keybindings editor was conservative: the `KeyBindings` core is shared and iPad supports hardware keyboards + SwiftUI keyboard shortcuts, so a remap UI is feasible. That framing is superseded; PREF-29 is the canonical entry. (Not listed as a hard-miss in `99`.)
