## Approach

Three reuse tiers drive every work item below:

1. **Already wx-free core — bridge + SwiftUI only.** The headline cases:
   - `SequenceElements::GetUsedColours(bool)` / `ReplaceColours(RenderContext*, from, to, selectedOnly)` (`src-core/render/SequenceElements.h:175-176`) — Color Replace needs zero core work.
   - `SequenceElements::SelectAllEffects()` / `SelectAllEffectsNoTiming()` / `UnSelectAllEffects()` (`SequenceElements.h:180-183`) — global Select-All.
   - The core `UndoManager` (`src-core/render/UndoManager.h`) already exposes `GetUndoString()` / `GetRedoString()` descriptive labels and the iPad-specific `SetMaxSteps()` cap.
2. **UI-layer logic on desktop — must be extracted to a core/bridge helper first.** `EffectsGrid::ConvertSelectedEffectsTo`, `FillRandomEffects`, `MoveSelectedEffectUp/Down` (`src-ui-wx/sequencer/EffectsGrid.h:115-116,142,282`), and the three `ShiftEffects*` variants (`src-ui-wx/xLightsMain.cpp:5797-5968`) live in wx UI code. These need a small extraction into a shared helper or a faithful Swift reimplementation over the bridge's per-effect mutators before the iPad can call them.
3. **Pure SwiftUI / view-model wiring.** Display Elements bulk-visibility / select / sort, reorder buttons, scope-delete menu entries, has-effects stripe, etc. — list and menu work in `DisplayElementsSheet.swift` / `RowHeaderViews.swift` against bridge accessors that already exist (`elementHasEffects`, `setElementVisible`, `rowIsModelGroupAtIndex`, `moveModel:inViewAtIndex:toPosition:`).

All mutating ops funnel through the single `@Observable SequencerViewModel` so undo registration + row reload happen in one place (existing convention — 26 `registerUndo` sites today). New touch-idiom UI uses sheets / long-press menus / inspectors rather than literal desktop dialog ports.

### The Undo/Redo decision (headline)

The iPad does **not** use `src-core/render/UndoManager` for the SwiftUI Cmd-Z path — it registers per-op inverse closures on Foundation `UndoManager` (`SequencerViewModel.undo()`:5234) and already calls `setActionName(...)` at ~26 sites with descriptive labels ("Delete N Effects", "Split Effect", etc.). The core `UndoManager` is still live but only drives bridge-internal ops (e.g. `ConvertEffectsToPerModel`), capped at 50 steps via `iPadRenderContext.cpp:73`.

**Recommendation: keep the Foundation-UndoManager architecture; do NOT unify onto core `UndoManager`.** Rationale: (a) the closure model already produces richer, more granular labels than the core's coarse `UNDO_*` markers; (b) SwiftUI grouping (`beginUndoGrouping`) maps cleanly onto Foundation and is woven through the view model; (c) unifying would mean re-expressing every Swift-side op as a core `UndoStep`, a large, regression-prone rewrite for no user-visible gain. The actual P0 work is two narrow fixes plus an audit (see **SEQ-1**), not an architecture change. We explicitly record this as a "feature iPad implements differently" rather than a gap to close by porting.

## Risks / open questions

- **Undo audit completeness (SEQ-1).** The closure-per-op model means any future mutating op that forgets `registerUndo` is silently non-undoable. Beyond the one-time audit, consider a lightweight convention/lint (e.g. a `mutate { }` wrapper in the view model that requires an inverse) so the gap can't reopen. Open question: is `toggleElementRenderDisabled` cheaply invertible, or does it need a captured before-state?
- **Core extraction blast radius (SEQ-9/11/26/30).** ShiftEffects, ConvertSelectedEffectsTo, FillRandomEffects, and AC draw all live in `EffectsGrid` (wx UI). Extracting them into shared core/bridge helpers benefits both platforms (per CLAUDE.md parity rule) but risks regressing desktop behavior — each needs the desktop path re-pointed at the extracted helper and re-verified, not just copied.
- **AC mode is genuinely XL (SEQ-30).** It's keyboard-and-cell-range centric on desktop; the touch redesign (drag-rectangle paint) is net-new UX, not a port. Recommend a dedicated design pass and treating it as a separate workstream — do not let it block the rest of Phase 2/3.
- **`AdjustSettingsAfterSplit` correctness (SEQ-5).** Need to confirm every effect type implements a sane split adjustment; some may no-op, in which case the current identical-copy behavior is already correct for them and we only fix the value-curve/position-dependent cases.
- **Display Elements sort comparators (SEQ-13).** The acceptance count is pinned at **11 strategies** (12 with the conditional master-view option) to match desktop `ViewsModelsPanel` exactly; the size/node-count "Groups At Top by …" variants and the controller/port/start-channel sorts need model metadata accessors (node count, controller/port, start channel) on the bridge that may not all exist yet — confirm coverage before committing the L estimate.
- **Convert To Effect core extraction (SEQ-33).** `RampLenColor`/`isOnLineColor`/`DoConvertDataRowToEffects` live in `tabSequencer.cpp` (wx UI). Lifting the ramp helpers into core and constructing per-strand `SingleLineModel`s must be re-pointed and re-verified on the desktop path too (parity rule), and depends on rendered `SequenceData` being available at conversion time on iPad.
- **Effect Presets persistence (SEQ-25)** overlaps the Jukebox/Presets theme; coordinate ownership of the rgbeffects preset-tree serialization so it isn't built twice.
