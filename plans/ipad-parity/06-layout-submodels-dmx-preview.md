# iPad Parity Plan 06 — Layout: Submodels / Faces / States, DMX / Moving Heads / 3D, Preview & View Objects

_Scope: gap-analysis areas **8** (Submodels / Faces / States / Aliases), **9** (DMX / Moving Heads / 3D Objects), and **10** (Preview, Background & View Objects)._

## Current parity

- **Area 8 — Submodels / Faces / States / Aliases (43 total: 9 implemented, 12 partial, 22 missing).** This is the weakest layout sub-area. SubModels are strong (full list editor, ranges grid, sub-buffer, generate-slices, reverse/sort, node-chase, aliases, and an interactive Metal tap-toggle + marquee node-select preview). Faces and States share a single **generic flat key/value editor** (`FaceStateEditorSheet`) with no structured phoneme/eyes/state grid, no type awareness, no image picker, no color well, and **no preview/node selection** — usable for editing pre-existing definitions but far from authoring parity. Import/export/output-to-lights/node-layout report are absent across all three editors.
- **Area 9 — DMX / Moving Heads / 3D Objects (38 total: 25 implemented, 9 partial, 4 missing).** Strong for *editing* (every DMX type is fully editable through the descriptor-driven per-type panel) but has a structural hole in *creation*: you cannot add a DMX model from the Add-Model picker even though core already supports it. GDTF/LOR import is absent. Remaining gaps are detail-level per-mesh / per-servo-image transform sub-properties and servo pivot params.
- **Area 10 — Preview, Background & View Objects (54 total: 35 implemented, 9 partial, 8 missing, 2 out-of-scope).** Surprisingly strong — full 2D/3D preview, cameras, viewpoints, background editing, all five view-object types, handles, align/distribute. Genuine gaps are long-tail: rename/delete a named preview, group-center crosshair, **wiring-diagram view**, real-world dimension readouts, virtual-canvas/grid-spacing setters, DXF export.

## Approach

The shared `src-core/` already implements every algorithm needed here. Across all three areas the work is **SwiftUI views + ObjC++ bridge wrappers** over existing core ops, funneled through the single `@Observable SequencerViewModel` (which centralizes undo registration + row/canvas reloads):

- **Core reuse is near-total.** `Model::SetFaceInfo` / `SetStateInfo` / `GetFaceInfo` / `GetStateInfo`, `SubModel` node-range machinery, `Model::ChannelLayoutHtml` (wx-free), `ModelManager::CreateDefaultModel` (already creates `Dmx*`/`Moving Head`/`Servo`/`Skull`/`Floodlight`), `Mesh`/`DmxImage`/`Servo` transform getters/setters, `RulerObject::ConvertDimension`, `BaseObject::SetCenterOffset`, and the wiring arg on `Model::DisplayModelOnWindow` all already exist and are linked.
- **The single highest-leverage move** is to **reuse `SubmodelPreviewPane`** (the existing Metal tap-toggle + two-finger marquee + `nodeNearPoint` / `nodes(in:)` node-select machinery) as a shared node-assignment component, then host it inside new structured Faces and States editors. This converts the biggest cluster of P1 gaps (areas 8) into a refactor-and-rehost rather than net-new picking math.
- **Touch idioms over literal ports.** Structured grids become SwiftUI `Form`/`Grid` with inline `ColorPicker` (hex helpers `dmxHexColor`/`dmxHexFromColor` already in `LayoutEditorView.swift`) and Files/Photos pickers; desktop's right-click row menus become long-press context menus and swipe actions; the Channel Layout report renders in a `WKWebView`/share sheet; output-to-lights is a toggle that pushes a channel frame over the already-wired networked `OutputManager` (sACN/ArtNet/DDP — USB/serial DMX has no iOS path and is excluded).
- **Bridge convention.** New methods go on `XLSequenceDocument` with `NS_SWIFT_NAME(…)`, matching the existing `faceInfoForModel`/`setFaceInfo`/`submodelDetailsForModel` surface. DMX per-type fields extend the existing descriptor `Build*Props`/`setPerTypeProperty` pattern in `XLSequenceDocument.mm` rather than adding new sheets.

Implemented rows (e.g. SubModel editor, DMX per-type panels, 2D/3D preview, cameras, viewpoints, view-object types) are the platform we build on and need no work.

## Quick wins (Rollup C — cheap, high-leverage early)

- **LAY-1 (S)** — Add DMX model from picker. Core `CreateDefaultModel` already supports every DMX type; this is a literal string-list + label addition that unlocks the entire (already-built) DMX editing surface for new models. Do first.
- **LAY-9 (S)** — Wiring view. Flip a hardcoded `false` to a toggle-bound bool on `DisplayModelOnWindow`; closes the wiring-view gap in two areas at once.
- **LAY-13 (S)** — SubModel buffer styles from `SubModel::GetBufferStyles()` instead of a static array; removes drift risk, one bridge call.
- **LAY-20 (S)** — Make virtual-canvas size + grid-spacing editable; the fields already render read-only and the code comments literally say "needs bridge setters."
- **LAY-8 (S)** — Servo pivot offsets; additive descriptor fields over existing `Servo::SetScaledPivotOffset*`.
- **LAY-6 (S)** — Faces/States Copy & Rename; pure Swift map manipulation over the existing round-tripping `setFaceInfo`/`setStateInfo`.

## Out of scope for this theme

- **SpaceMouse / 3Dconnexion 6-DOF navigation (area 10)** — no 3Dconnexion HID driver or public 6-DOF API on iPadOS; multi-touch + Pencil are the native equivalents.
- **Double-click to maximize preview pane (area 10)** — iPadOS windowing is OS-managed (Stage Manager / full-screen scenes); double-tap is repurposed for camera reset.

## Risks / open questions

- **Faces/States schema fidelity (LAY-3/4/5).** The desktop FaceStateData map encodes phoneme/eyes/outline keys, `Type`, `CustomColors`, and per-element `*-Color` keys in a specific layout. The round-trip works today only because the iPad treats it as opaque; a structured editor must reproduce the *exact* key naming/ordering or it will silently desync with desktop-authored or rgbeffects-authored definitions (note: `EffectCustomRowsFinal.swift:768` already requires `Type=SingleNode` + `CustomColors` for the DMX-state effect path). Validate round-trips against desktop-saved models before shipping.
- **Reusing `SubmodelPreviewPane` for Faces/States (LAY-2).** The submodel preview highlights the *active line*; faces/states need multiple distinctly-colored elements visible at once. Decide early whether LAY-2 supports a multi-element color overlay or only single-active-element highlight, since LAY-3/4/5/10/12 all depend on it.
- **Output-to-lights scope (LAY-12).** Networked output (sACN/ArtNet/DDP) is feasible via the existing `OutputManager` bridge, but USB/serial DMX has no iOS path — the UI must make the networked-only limitation clear rather than appear broken when no networked controller is reachable.
- **GDTF zip extraction (LAY-24).** Needs a zip-extract step before `GdtfParser`; confirm an available unzip path in the iPad lib (zstd is present, but GDTF is a standard zip) and where extracted assets land in the sandbox.
- **Effort honesty.** LAY-3 and LAY-4 are the two largest single items (structured grid + color + node-select host); they are the core of closing the area-8 weakness and should be scheduled as deliberate L efforts, not lumped with the quick wins.
- **3D node-assignment.** The submodel preview node-pick is documented as a 2D pick; faces/states authoring inherits that limitation, which is acceptable but should be stated to users.
- **Multi-select body-drag undo granularity (LAY-33-touch).** Desktop creates one undo point per moved model during the drag (it calls `CreateUndoPoint` for each selected model in the move loop). The iPad must decide whether to coalesce the multi-model move into a single undo step (preferred touch idiom) or mirror desktop's per-model points; either way, locked models must be skipped (desktop guards on `IsLocked()`), and view-objects selected alongside models should move too.
- **Position-zones pref gating (LAY-31).** The editor and its preview overlay are governed by the `EnablePositionZones` / `ShowZoneIndicator` prefs — confirm those prefs are exposed (or default sensibly) on iPad before surfacing the editor, or the feature appears dead.
