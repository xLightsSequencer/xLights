## Current parity

Both areas are already strong. **Area 12** is 50 features: 33 implemented, 9 partial, 7 missing, 1 out-of-scope — roughly 70-75% to parity, with the entire Metal waveform stack (peak+RMS, zoom/scroll/playhead), every band/stem/vocal filter, onset/pitch/spectrogram overlays, sound classification, HTDemucs stems, and the lyric breakdown/auto-label/subdivide/create-from-effects pipeline already wired through shared `src-core` media + lyrics code. **Area 17** is 42 features: 21 implemented, 9 partial, 8 missing, 4 out-of-scope — transport, alternate-track CRUD, the Media tab (embed/extract/replace/rename), and audio-driven timing generation are all done. The remaining work in both areas is overwhelmingly SwiftUI + ObjC++ bridge wiring over ops that **already exist in shared core**, plus a few genuine iOS platform constraints (VAMP dylib loading, video transcode/encode, audio output-device enumeration).

## Approach

- **Shared core is already linked.** Almost nothing here is a new algorithm. The detectors (`OnsetDetector`/`TempoDetector`/`ChordDetector`/`PitchDetector`/`SoundClassifier`/`StemSeparator`), `AudioManager`, `SequenceMedia`, `PhonemeDictionary`, and the timing parsers all live in `src-core` and the iPad already calls many of them. Each work item below names the **exact shared-core call or existing bridge method** so the task is "add a SwiftUI affordance + (sometimes) a thin `NS_SWIFT_NAME` wrapper," not "port logic."
- **Bridge surface.** `XLSequenceDocument` already exposes `importXTimingFromPath:`, `importLorTimingFromPath:`, `importPapagayoTimingFromPath:`, `exportTimingTrackAtRow:toPath:`, `addFixedIntervalTimingTrackNamed:intervalMS:`, `breakdownPhraseAtRow:atIndex:`, `breakdownWordsAtRow:`, and `removeWordsAndPhonemesAtRow:`. New wrappers needed are small and follow the existing `NS_SWIFT_NAME(…)` convention.
- **Touch idioms over literal ports.** Replace desktop modal dialogs with SwiftUI sheets/inspectors and long-press menus. The Add Timing Track sheet, mark long-press menu, and row-header context menu already exist as the host surfaces; we extend them rather than introduce new chrome.
- **Single funnel.** All mutating ops route through `SequencerViewModel` so undo registration + row reload stay in one place (matching the existing `breakdownPhrases`/`halveTimingMarks` pattern).
- **Don't fight iOS.** VAMP plugin hosting, raw video transcode, and audio output-device enumeration stay out of scope (App Store / platform constraints) — the native analyzers are the accepted substitute.

## Out of scope for this theme

(One line each; full reasons in the master out-of-scope doc, Rollup B.)

- **VAMP plugin beat/bar/key analysis & "Download VAMP plugins" prompt** — App Store forbids loading external Queen Mary `.dylib` plugins at runtime; on-device Apple analyzers (onsets/tempo/chords/pitch/SNClassify) are the substitute.
- **VAMP plugin timing-track generation** — same dylib-host constraint (`xLightsVamp.h` is gated desktop-only).
- **Polyphonic transcription** — depends on the VAMP polyphonic plugin (`DoPolyphonicTranscription` routes through VAMP), so the file-based Note Import in TIM-7 ships without it.
- **Incompatible-video conversion on import** — out of scope for a *decode* reason, not encode: `VideoTranscoder` exists to handle sources VideoToolbox can't decode, and its FFmpeg **decode** half is desktop-only. (Its *encode* half now runs through the shared-core AVFoundation `VideoWriter`, but iPad can't decode the inputs that need transcoding, so the path is still moot.) iPad detects + warns (MED-3) rather than transcodes.
- **Audio output device selection** — iOS routes audio via the OS (AVAudioSession / Control Center / AirPlay); there is nothing for an app-side device picker to enumerate.
- **Export rendered sequence as video** — ⬆ **encoder now in shared core (2026-06):** `VideoExporter` was replaced by `src-core/media/VideoWriter` (AVFoundation H.264/H.265/ProRes/uncompressed), which builds in the iPad lib. The House-Preview video export is now a wiring task (see 09-file-lifecycle-render-tools). The *model* video export (`WriteVideoModelFile`) additionally needs its `wxImage`/`FillImage` frame production moved off wx before it can run on iPad; the encoder half is done.

## Risks / open questions

- **Vixen3 / LSP parsers are wx-side (TIM-2).** `Vixen3.cpp` lives under `src-ui-wx/import_export/`. Confirm whether the parse body is wx-free enough to relocate to `src-core` (matching where `ProcessSRT`/`ProcessElevenLabs` already live) or whether it must be deferred. SRT/Audacity/ElevenLabs are confirmed shared core; LOR/Papagayo bridges are confirmed present. Land the confirmed formats first, defer LSP/Vixen3 if the move isn't cheap.
- **Note Import parser relocation (TIM-7).** `LoadAudacityFile`/`LoadMIDIFile` are currently `xLightsFrame` methods. Moving them to core must shed any wx types (`wxString`, `wxFileName`) — quick audit needed before committing to L effort.
- **Undo consistency.** The iPad uses Foundation `UndoManager` per-op closures, not core `UndoManager`. Every new mutating op (TIM-1/3/4/5, MED-1/2/5) must register an undo closure + trigger a row reload through `SequencerViewModel`, consistent with existing timing ops, or it will silently break undo.
- **Multi-format dispatch by extension is ambiguous** for `.txt` (Audacity labels vs other) and `.json` (ElevenLabs vs unrelated). Decide whether to sniff content or let the format-specific parser fail gracefully and surface an import-error alert (the existing import-error alert path can carry the message).
- **`-ffast-math` in the on-device analyzers** — any new detector glue in `src-core` must use `xl::isnan/isinf` and `numeric_limits::max()` sentinels per the project fast-math rules; this only matters if TIM-7 adds new core parse math.
- **Render-time video decode failures have no fallback (MED-3 / R-1).** The iPad lib ships no FFmpeg, so codecs VideoToolbox can't decode (VP9, AV1, ProRes-RAW, WMV) — and the known H.264 High 4:4:4 below-64-px `-8969` case — fail only when the Video effect renders, not at import. Decide how the bridge propagates the decode error so the UI alerts rather than rendering black; the import-time `videoCompatibilityIssueForPath:` check does not catch all of these.
- **`.xfacemap` / Generate-Lyrics-From-Data** intentionally deferred — confirm with product that the legacy singing-faces reconstruction workflow has no iPad demand before fully dropping it. (Note this is the face-map reconstruction path, not the audio-derived AI Speech-to-Lyrics feature.)
- **MIDI Type-0 / packed files (TIM-7).** The iPad parser (relocated `LoadMIDIFile` or AVFoundation `MIDIFile`) must replicate desktop's #6434 fix: a Type-0 single-track merged-channel file must read events from the joined track, or it imports zero marks. Verify whichever parser path is chosen handles both Type-0 and Type-1 before settling on AVFoundation vs the relocated wx parser.
